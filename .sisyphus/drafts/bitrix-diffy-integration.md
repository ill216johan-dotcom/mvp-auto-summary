# Draft: Интеграция Битрикс24 CRM → Diffy

## Требования (подтверждённые)
- Подтягивать из Битрикса в Diffy: **звонки**, **письма (с телом!)**, **комментарии сотрудников**
- Всё привязывается к **лидам** в Diffy
- Приоритет выше, чем интеграция с Jira
- REST API webhook: `https://bitrix24.ff-platform.ru/rest/1/fhh009wpvmby0tn6/`
- Доступы: CRM, телефония, пользователи
- Связь лидов: через пользовательское поле в Битриксе с номером договора (ФФ-1000139, ФФ-4405)
- Частота синхронизации: **раз в день** (загрузка новых данных)
- Данные из звонков: **метаданные + транскрипция** (если доступна)
- **Данные из писем: ПОЛНЫЙ ТЕКСТ + метаданные** (тема, от/кому, дата, ТЕЛО ПИСЬМА) — критично для претензий клиентов
- Комментарии: **любые текстовые записи** сотрудников (заметки на таймлайне, описания задач)
- Воронка: Лид (отдел продаж) → Контакт (заезд на фулфилмент, регистрация в ВМС, номер договора)
- **ГЛАВНЫЙ ПЛАН**: лиды (без договоров) тоже должны иметь свои "блокноты"
- Лиды без договоров: **нужно создать из Битрикса** (в Diffy их пока нет)
- Хранение: **PostgreSQL + Dify KB** (для RAG-поиска по коммуникациям)
- Саммари: **да, нужны** — Claude обрабатывает звонки+письма+комменты за день → сводка по лиду

## Архитектура двух блокнотов (CRITICAL)

### Битрикс CRM: две сущности на клиента
```
Битрикс "Лид" (ID=123, без договора)
  ├─ Коммуникации: звонки, письма, комменты (период работы отдела продаж)
  ├─ Статус: NEW → PROCESSED → ...
  └─ Ответственный: менеджер отдела продаж

       ↓ (клиент заинтересовался, заезд, регистрация)

Битрикс "Контакт" (ID=456, с номером договора ФФ-4405)
  ├─ Коммуникации: звонки, письма, комменты (период после регистрации)
  ├─ Номер договора: ФФ-4405 (в пользовательском поле UF_*)
  └─ Ответственный: менеджер по работе с клиентом

ВАЖНО: Старые данные (из периода "Лида") ОСТАЮТСЯ привязаны к сущности "Лид" (ID=123).
        Новые данные (после регистрации) идут в сущность "Контакт" (ID=456).
        Это ДВЕ РАЗНЫЕ записи в CRM Битрикса.
```

### Diffy: ДВА блокнота на клиента
```
Блокнот 1: BX-LEAD-123 (период лида, отдел продаж)
  ├─ Источник: Битрикс "Лид" ID=123
  ├─ Данные: звонки, письма, комменты из периода "лида"
  ├─ Dify KB: отдельный dataset для истории продаж
  └─ Саммари: "Работа отдела продаж с лидом"

Блокнот 2: LEAD-4405 (период контакта, фулфилмент)
  ├─ Источник: Битрикс "Контакт" ID=456
  ├─ Номер договора: ФФ-4405
  ├─ Данные: звонки, письма, комменты из периода "контакта"
  ├─ Dify KB: существующий dataset (уже есть в Diffy)
  └─ Саммари: "Работа с клиентом после заезда"
```

### Синхронизация из обеих сущностей Битрикса
- **Лиды Битрикса** (OWNER_TYPE_ID=1): создаём BX-LEAD-{bitrix_id} в Diffy
- **Контакты Битрикса** (OWNER_TYPE_ID=3): создаём/обновляем LEAD-{contract_number} в Diffy

## Технические решения
- Webhook авторизация (не OAuth) — достаточно для MVP, бессрочный ключ
- Пагинация: 50 записей за запрос, цикл по `next`
- Batch-запросы: до 50 команд в одном HTTP-запросе для оптимизации
- Лимиты: ~5000 запросов/час (webhook)
- **Синхронизация И Лидов И Контактов**: crm.activity.list с OWNER_TYPE_ID=1 (Лид) + OWNER_TYPE_ID=3 (Контакт)

## Результаты исследования

### Проект Diffy (mvp-auto-summary)
- Python 3.11, PostgreSQL 16, APScheduler, Docker Compose
- Лиды хранятся в `lead_chat_mapping` (lead_id, lead_name, chat_id, dify_dataset_id, curators)
- Данные: processed_files (транскрипции), chat_messages (Telegram), client_summaries
- Интеграции: Telegram Bot, Dify.ai (RAG), Claude/z.ai, Whisper
- **Нет существующей интеграции с Битрикс24**
- Формат lead_id: "LEAD-4405", "LEAD-987" и т.д. (только контакты с договорами)
- Ключевые файлы: app/main.py, app/core/db.py (496 строк SQL), app/scheduler.py
- Scheduler jobs: scan_recordings (каждые 5 мин), daily_digest (23:00), individual_summary (22:00)

### Bitrix24 REST API
- `crm.lead.list` — получение лидов (OWNER_TYPE_ID=1) с фильтрацией и пагинацией (50/стр)
- `crm.contact.list` — получение контактов (OWNER_TYPE_ID=3) с номерами договоров
- `crm.lead.fields` / `crm.contact.fields` — получение всех полей (включая UF_*)
- `crm.activity.list` — активности лида/контакта (параметр OWNER_TYPE_ID: 1=Lead, 3=Contact)
- Типы активностей: CALL (TYPE_ID=1), EMAIL (TYPE_ID=4), MEETING (TYPE_ID=2)
- `voximplant.statistic.get` — история звонков с CALL_RECORD_URL, CALL_DURATION, PHONE_NUMBER
- `crm.timeline.comment.list` — комментарии сотрудников на таймлайне
- `user.get` — данные сотрудников по ID (NAME, LAST_NAME, WORK_POSITION)
- Batch: до 50 команд за запрос
- Фильтры: =, !=, >, <, >=, <=, %, @ (IN)

### Ключевые находки
- Звонки: хранятся и в crm.activity (TYPE_ID=1) И в voximplant.statistic
- Записи звонков: поле SRC_URL/CALL_RECORD_URL в voximplant.statistic
- Транскрипция: делаем сами через Whisper (не полагаемся на Битрикс)
- **Emails**: TYPE_ID=4 в crm.activity, поля SUBJECT, DESCRIPTION (BODY — полный текст HTML), DIRECTION
- Пользовательские поля: UF_* — нужно узнать имя поля с номером договора через crm.contact.fields

## Все вопросы закрыты

### Решения приняты
- **Тело писем**: ВКЛЮЧЕНО (DESCRIPTION/BODY field) — критично для претензий клиентов
- **Двойная сущность**: ДВА блокнота на клиента (BX-LEAD-123 + LEAD-4405)

- **Синхронизация обеих сущностей**: И Лиды (OWNER_TYPE_ID=1) И Контакты (OWNER_TYPE_ID=3)
